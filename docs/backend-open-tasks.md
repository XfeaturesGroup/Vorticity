## 🔴 Критично — реальная брешь, не просто «незаконченная фича»

- [x] **`POST /token/issue` — issuance-оракул без аутентификации.** `workers/enrollment/src/index.ts`
  подписывает любое присланное `blinded`-сообщение без проверки, что вызывающий вообще прошёл
  `/oauth/callback`. Любой может дергать эндпоинт напрямую и штамповать неограниченное число анонимных
  redemption-токенов — весь смысл "один Xfeatures-аккаунт → ограниченное число входов" (R11, G8) декоративен.
  **[ЗАКРЫТО 2026-07-23]** — одноразовый HMAC-тикет из `/oauth/callback` (`ticket.ts`), обязателен и
  тратится в `/token/issue`. Живой тест: 401 на отсутствующий/битый/просроченный/подделанный тикет, 200→409
  на повтор, полный RSABSSA-цикл не сломан, 7/7 в `packages/ohttp` live E2E. См. `docs/06` R11 progress-note.

## 🟠 R11 — Sybil quota по PPID (телеметрия есть, порога нет)

- [x] `enroll_count` в `enroll_ppid` растёт бесконечно, нигде не проверяется потолок.
  **[ЗАКРЫТО 2026-07-23]** — та же механика тикета: `/oauth/callback` проверяет `ENROLL_QUOTA_MAX=20`
  ДО апсерта. Живой тест через локальный mock IDM: 20 успешных `/oauth/callback` на один синтетический
  аккаунт, 21-й → 403; `enroll_count` в D1 остановился ровно на 20.

## 🟡 R8 / Sealed Sender++ — поле отправителя не шифруется

- [x] `packages/vortic-core/src/seal.rs` — `seal()`/`unseal()` были `todo!()`, никем не вызывались.
  **[ЗАКРЫТО НА УРОВНЕ ПРИМИТИВА, 2026-07-23]** — реальный ephemeral-X25519-DH-to-recipient +
  ChaCha20-Poly1305, sender_cert зашифрован внутри огибающей, паддинг до `2^bucket` байт (математика
  бакетов зеркалит `bucketing.ts`). 8 новых native-тестов, cargo 82/84 (2 непричастных к этому пассу
  падения — `openmls` и таймингозависимый PoW-тест). Оба build-профиля (`edge-verify-only`,
  `client-full` wasm32) собираются чисто. **Честно не сделано в этом пассе:** проводка
  `seal_message`/`unseal_message` в реальный send/receive-путь (`useQueueTransport.ts`, ratchet-конверт)
  — это отдельный follow-up, чтобы не раздувать объём; cover traffic (K6) не тронут.

## 🟢 R10 — нет backpressure на QueueDO/ConvLogDO/GroupDO

- [x] Поток пушей мог разогнать одну DO к лимиту 10GB без всякой защиты.
  **[ЗАКРЫТО 2026-07-23]** — defensive cap (QueueDO: 10k msg/64MiB; ConvLogDO/GroupDO: 500k
  entries/256MiB), 429/507 при превышении. Живой тест (временно занижал константы): все 3 DO дали
  ровно ожидаемые коды на границе. Заодно найден и починен реальный баг — `GroupDO.webSocketClose`
  был без try/catch (третий случай этого же паттерна в проекте, у QueueDO/ConvLogDO уже был пофикшен).

## 🟡 Прочее реально открытое (проверено по коду, не по доку)

- [x] **Cover traffic (K6)** — R7/R8. **[ПРОВЕРЕНО, БЭКЕНД-ЗАДАЧИ НЕТ]** — транспорт (`QueueDO`/
  `ConvLogDO`/`GroupDO`) уже структурно не отличает decoy-сообщение от настоящего (это и есть
  требуемое свойство), так что "поддержка" cover traffic со стороны бэкенда не нужна — сервер уже
  готов, ничего доделывать. Вся фича целиком клиентская: КОГДА слать decoy и как получатель отличает
  и отбрасывает его после расшифровки — это `apps/web`, вне рамок этой сессии, не полуразобранный
  бэкенд-таск.
- [x] **MLS group ciphersuite не PQ** — R7.
  **[ЗАКРЫТО 2026-07-23]** — при повторной проверке оказалось, что это НЕ апстрим-блокер, а
  двухстрочная недостающая проводка в `openmls_rust_crypto` 0.5.1's `kem_mode()` (буквальный
  `unimplemented!()`), при том что сам примитив X-Wing уже полностью реализован в
  `hpke-rs-rust-crypto` (уже была наша зависимость). Завендорил и запатчил `openmls_rust_crypto`
  (`packages/vortic-core/vendor/openmls_rust_crypto/`, MIT, честно и заметно помечен как
  maintenance liability с инструкцией убрать при апстрим-фиксе) — 2 match-arm правки, никакой новой
  криптографии. Второй реальный баг, найденный только живым прогоном: `hpke-rs-rust-crypto`'s
  X-Wing спрятан за собственным cargo-feature `experimental` — без него тихий `CryptoLibraryError`
  без объяснения причины. `group.rs`'s `CIPHERSUITE` теперь настоящий
  `MLS_256_XWING_CHACHA20POLY1305_SHA256_Ed25519` — группы теперь реально постквантовые (гибрид
  ML-KEM-768+X25519), как и 1:1-ветка. Живой тест: все 5/5 тестов `group.rs` зелёные в `--release`
  (единственный debug-режимный "провал" — тот же самый предсуществующий и уже задокументированный
  `debug_assert!`-баг в самом openmls, не регрессия); полный набор 89/89 в release;
  `edge-verify-only` подтверждён через `cargo tree` (ноль openmls/x-wing/ml-kem в графе); реальный
  `wasm-pack build:client`/`build:msg`/`build:issuer` пересобраны чисто; полный `packages/ohttp`
  live E2E набор 7/7 на пересобранных wasm-бандлах. Честно не сделано: `apps/web`-проводка для MLS
  групп, доставка Welcome через `QueueDO` (отдельный, нетронутый пункт R7).
- [x] **`oprf.rs` — мёртвый код или незаконченный DLEQ.** **[ПРОВЕРЕНО, РЕШЕНИЕ УЖЕ ПРИНЯТО РАНЕЕ]** —
  это не открытый вопрос: ещё в "Plane Bridge" проходе явно решили оставить модуль (реальный,
  протестированный код), просто не вырывать целый крипто-модуль ради замены схемы. Не переоткрываю.
- [x] **R4/R21 doc-sync gap** — **[ЗАКРЫТО 2026-07-23]** — добавлена progress-note к R4, сославшаяся на
  уже существующий официальный ceremony key из R21-continued. Только документация, кода не было.
- [x] **R12 — forward-secret key rotation для backup.**
  **[ЗАКРЫТО 2026-07-23]** — индексированный (не цепочный!) HKDF по публичному номеру поколения:
  `gen_key[n] = HKDF(master_key, "vortic-backup-gen-v1", n)`. Не цепочка специально: цепочка дала бы
  утечке одного поколения все последующие ключи бесплатно. `n` публичный — сервер хранит его в R2
  custom metadata, восстановление узнаёт `n` у сервера и просто пересчитывает по фразе, без
  локального состояния. AEAD AAD = `backup_id‖n` — доп. рубеж поверх вывода ключа. На сервере —
  настоящий atomic CAS (`onlyIf.etagMatches`), не read-then-write: счётчик обязан расти ровно на 1,
  иначе 409. Старые `backup_export`/`backup_import` без поколений удалены (не было живых вызывающих).
  22/22 cargo-теста, живой тест CAS-последовательности через wrangler dev, регрессия — весь ohttp
  live E2E набор 7/7. Честно не покрыто: `apps/web` (не было и не будет в рамках этого прохода),
  первая запись (bootstrap) без строгого CAS (узкий редкий кейс), и подмена сервером устаревшей пары
  (ciphertext, n) именно во время ВОССТАНОВЛЕНИЯ без локальной памяти клиента — недетектируемо по
  дизайну, тот же класс остаточного риска, что и у любой облачной backup-схемы такого типа.
- [x] ~~**R14 backend-половина (WASM integrity pinning)**~~ — рассмотрел и ОТКЛОНИЛ как задачу: `.wasm`
  импортируется wrangler'ом уже СКОМПИЛИРОВАННЫМ (`WebAssembly.Module`), сырых байт для хэширования в
  рантайме нет; и даже если бы были — подмена произошла бы на этапе сборки того же бандла, где лежит и
  сама хэш-константа для проверки, так что рантайм-проверка не добавляет реального рубежа защиты. R14's
  "client-side circuit-hash pinning" в доке не случайно назван client-side — это защита файла, который
  браузер отдельно скачивает с CDN (`apps/web/public/zk/*`), а не бандла самого Worker'а. У бэкенда для
  ЭТОГО конкретного риска нет отдельной точки подмены после сборки — не заглушка вместо задачи, а честный
  вывод "здесь для бэкенда нечего чинить".

## ⚙️ Инфраструктурное/операционное (не код)

- **R18 KT-монитор** — код (`kt-monitor.mts`) реальный и работает, но никогда не запускался с независимо
  контролируемого аккаунта/машины — без этого gossip/detection не даёт настоящего trust property.
- **R26 cross-zone relay** — `workers/ohttp-relay` сейчас в том же Cloudflare-аккаунте, что и Gateway;
  платформенная cross-zone анонимизация `CF-Connecting-IP` недоступна без второго аккаунта. Блокирует выход
  за пределы доверенной альфа-группы.
